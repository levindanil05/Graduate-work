# 1. 🎯 Цель системы

Разработать веб-приложение для:

* централизованного хранения файлов учебных планов (PLX)

# 4. 🔄 Workflow (жизненный цикл)

Реализуется через **django-fsm**

## Статусы:

* new (новый)
* on_review (на проверке)
* needs_fix (на доработке)
* approved (утверждён)
* archived (архив)

---

## Переходы:

| Откуда                | Куда               | Кто           |
| --------------------- | ------------------ | ------------- |
| new → on_review       | отправка           | кафедра       |
| on_review → needs_fix | замечания          | учебный отдел |
| needs_fix → on_review | повторная отправка | кафедра       |
| on_review → approved  | утверждение        | учебный отдел |
| approved → archived   | автоматом          | система       |

---

## Требования:

* проверка прав при переходе
* логирование переходов
* отображение истории статусов

# 🧱 1. Общая архитектурная идея

Стиль: **modular monolith (Django apps + сервисный слой)**
Принципы:

* зависимости только **через интерфейсы (ports)**
* доменная логика не знает про Django ORM напрямую (по возможности)
* файловая система, workflow, парсер — **плагины**

---

# 🧩 2. Подсистемы (границы ответственности)

## 2.1 📦 Catalog (реестр учебных планов)

**Что делает:**

* хранит сущность StudyPlan
* агрегирует текущую версию и метаданные
* отдаёт данные для таблицы

**Не знает про:**

* как хранятся файлы
* как работает workflow
* как парсится PLX

---

## 2.2 📁 File Storage (абстракция хранилищ)

На базе pyfilesystem2

**Что делает:**

* загрузка / скачивание файлов
* работа с путями
* синхронизация зеркал

**Контракт:**

```python
class StoragePort(Protocol):
    def save(file: BinaryIO, path: str) -> str
    def open(path: str) -> BinaryIO
    def delete(path: str) -> None
    def exists(path: str) -> bool
    def list(path: str) -> list[str]
```

---

## 2.3 🧾 Versioning (версии файлов)

**Что делает:**

* создаёт новую версию
* хранит историю
* связывает файл + метаданные + комментарий

**Контракт:**

```python
class VersionService:
    def create_version(plan_id, file, author, comment) -> VersionDTO
    def get_versions(plan_id) -> list[VersionDTO]
    def get_file(version_id) -> FileRef
```

---

## 2.4 🧠 Metadata Extraction (PLX parser)

**Что делает:**

* извлекает метаданные из файла

**Контракт:**

```python
class MetadataExtractor(Protocol):
    def extract(file: BinaryIO) -> dict
```

➡️ Важно: это **чистый модуль**, без Django

---

## 2.5 🔄 Workflow (жизненный цикл)

На базе django-fsm

**Что делает:**

* управляет статусами
* валидирует переходы
* проверяет права

**Контракт:**

```python
class WorkflowService:
    def get_status(plan_id) -> str
    def can_transition(plan_id, user, action) -> bool
    def transition(plan_id, user, action) -> None
    def get_available_actions(plan_id, user) -> list[str]
```

---

# 🔗 3. Как они взаимодействуют (главный сценарий)

## 📥 Загрузка новой версии

```id="0rqy5e"
User → API → VersionService.create_version()

VersionService:
    → Storage.save()
    → MetadataExtractor.extract()
    → DB.save(version)
    → Catalog.update_current_version()
    → EventBus.publish("version_created")
```

---

## 🔄 Изменение статуса

```id="kshbwx"
User → API → WorkflowService.transition()

WorkflowService:
    → проверка прав
    → изменение статуса
    → EventBus.publish("status_changed")
```

---

## 💬 Комментарий

```id="5b2sqp"
User → API → CommentService.add_comment()
```

# 🧠 4. Ключевой слой — Application Services

Это “склейка”:

```python
class PlanApplicationService:

    def upload_new_version(...):
        version = VersionService.create_version(...)
        WorkflowService.transition(..., action="submit")

    def approve_plan(...):
        WorkflowService.transition(..., action="approve")
```

➡️ Здесь бизнес-логика
➡️ Подсистемы не знают друг о друге

---

# 🔌 5. Внедрение зависимостей (важно)

Вместо жёсткой связки:

```python
storage = LocalStorage()
```

Используем:

```python
def get_storage() -> StoragePort:
    return settings.STORAGE_BACKEND
```

➡️ можно заменить:

* локальное хранилище
* WebDAV
* S3

# 🧩 6. Где слабое зацепление реально достигается

| Подсистема | Независимость         |
| ---------- | --------------------- |
| Metadata   | ✅ полностью           |
| Storage    | ✅ полностью           |
| Workflow   | ⚠️ зависит от модели  |
| Versioning | ⚠️ зависит от Storage |
| Comments   | ✅                     |
| Query      | ⚠️ зависит от ORM     |

---

# ⚠️ 7. Где НЕ получится полностью развязать

Честно:

* Django ORM всё равно связывает модели
* workflow привязан к модели статуса
* Query слой зависит от структуры БД

➡️ это нормально для modular monolith
