# 4. 🔄 Workflow (жизненный цикл)

## Статусы:

* archived (архив)

*Актуальные определения и полный словарь статусов — в разделе **10.7**; первая загрузка, ошибка парсера, корзина и purge — в **10.15**.*

# 🧱 1. Общая архитектурная идея

Стиль: **modular monolith (Django apps + сервисный слой)**
Принципы:

* зависимости только **через интерфейсы (ports)**
* доменная логика не знает про Django ORM напрямую (по возможности)
* файловая система, workflow, парсер — **плагины**

# 🧩 2. Подсистемы (границы ответственности)

## 2.1 📦 Catalog (реестр учебных планов)

**Что делает:**

* хранит сущность StudyPlan
* агрегирует текущую версию и метаданные
* отдаёт данные для таблицы

**Не знает про:**

* как хранятся файлы
* как работает workflow
* как парсится PLX

---

## 2.2 📁 File Storage (абстракция хранилищ)

На базе pyfilesystem2

**Что делает:**

* загрузка / скачивание файлов
* работа с путями
* синхронизация зеркал

**Контракт:**

```python
class StoragePort(Protocol):
    def save(file: BinaryIO, path: str) -> str
    def open(path: str) -> BinaryIO
    def delete(path: str) -> None
    def exists(path: str) -> bool
    def list(path: str) -> list[str]
```

---

## 2.3 🧾 Versioning (версии файлов)

**Что делает:**

* создаёт новую версию
* хранит историю
* связывает файл + метаданные + комментарий

**Контракт:**

```python
class VersionService:
    def create_version(plan_id, file, author, comment) -> VersionDTO
    def get_versions(plan_id) -> list[VersionDTO]
    def get_file(version_id) -> FileRef
```

---

## 2.4 🧠 Metadata Extraction (PLX parser)

**Что делает:**

* извлекает метаданные из файла

**Контракт:**

```python
class MetadataExtractor(Protocol):
    def extract(file: BinaryIO) -> dict
```

➡️ Важно: это **чистый модуль**, без Django

## 2.5 🔄 Workflow (жизненный цикл)

На базе django-fsm

```python
class WorkflowService:

```

# 🧠 4. Ключевой слой — Application Services

Это “склейка”:

```python
class PlanApplicationService:

    def upload_new_version(...):
        version = VersionService.create_version(...)
        WorkflowService.transition(..., action="submit")

    def approve_plan(...):
        WorkflowService.transition(..., action="approve")
```

➡️ Здесь бизнес-логика
➡️ Подсистемы не знают друг о друге

# 10. Зафиксированные решения (2026-05-11)

Этот раздел фиксирует принятые решения и имеет приоритет над более ранними формулировками в документе.

## 10.1 Документирование решений

- Все принятые архитектурные и продуктовые решения фиксируются в папке `docs`.
- Текущий файл используется как рабочий "источник истины" для "дорожной карты" и архитектурных оговорок.

---

## 10.2 Обобщённая модель документа (не только УП)

- Базовая сущность системы: **Document** (обобщённый документ), а не только StudyPlan.
- Учебный план — это тип документа: `document_type = "study_plan"`.
- Система должна поддерживать добавление новых типов документов без нарушения текущей логики.
- Для разных типов документов допускаются опциональные связи между документами (например, производный документ, связанный с учебным планом).
- Общие операции (загрузка версии, хранение истории, статусы, комментарии, экспорт, права) проектируются на уровне `Document`.

---

## 10.3 Дополнительные поля: гибридная стратегия хранения

- Поля, которые активно участвуют в сортировке/фильтрации, хранятся в отдельной таблице (индексируемой).
- Более пассивные и редко используемые поля хранятся компактно в JSON-поле в основной таблице документа.
- Для УП это означает гибрид:
  - индексируемые атрибуты -> отдельные структуры;
  - расширяемые/служебные атрибуты -> `extra_data` (JSON).
- Конкретный набор индексируемых полей не должен быть жёстко зашит и может меняться по мере эксплуатации.
- Для поля, которое хранится в JSON, не создаются записи в таблице данных, но создаётся одна запись в таблице описания полей (с указанием, что тип хранения -- непосредственно при описываемой записи - вроде inline/inner).

---

## 10.4 Идентификация документа по имени файла и алиасам

- Базовая идентификация УП строится вокруг специального имени файла (текущий производственный подход).
- Состав идентификатора (факультет, направление, форма, кафедра, год и т.д.) **не фиксируется жёстко в коде** как неизменяемый шаблон.
- Требуется поддержка нескольких форм одного идентификатора (алиасы), включая варианты после транслитерации и других преобразований.
- Нужен слой нормализации и сопоставления имён:
  - выделение канонического идентификатора;
  - хранение списка допустимых алиасов;
  - сопоставление загружаемой версии к существующему документу через канонизацию + таблицу алиасов.

---

## 10.5 Идемпотентность загрузок через хэш

- Повторная загрузка идентичного файла должна быть идемпотентной.
- Механизм: вычисление хэша содержимого и сравнение с последними версиями документа.
- Допустимый и предпочтительный вариант: `xxhash`.
- При совпадении хэша:
  - не создавать дублирующую версию;
  - возвращать результат как "already exists / duplicate upload" с указанием существующей версии;
  - дать пользователю возможность перейти к найденной версии;
  - при явном подтверждении пользователя разрешать "активировать как свежую" уже существующую версию (без повторной записи бинарного файла).

---

## 10.6 Именование файлов версий и политика хранения истории

- Актуальная версия файла хранится под исходным именем (без добавок/изменений).
- Старые версии именуются с добавлением суффикса `_v{дата}`.
- Формат суффикса уточняется по частоте изменений:
  - если изменение единственное за дату -> `_vYYYYMMDD`;
  - если в одну дату несколько изменений -> `_vYYYYMMDD-HHMM`;
  - если в одну минуту несколько изменений -> `_vYYYYMMDD-HHMMSS`;
  - если коллизия в ту же секунду -> добавляется порядковый постфикс в скобках, например `(2)`, `(3)`.
- История версий по умолчанию не удаляется.
- Предпочтительный вариант долговременного хранения: кастомная реализация архивирования старых версий в один сжатый архив.
- Для оптимизации хранения допускается эвристика высокой схожести версий (ориентир: 90%+ похожести) при выборе стратегии упаковки.
- Поведение «распакованная область + архив» относится к **продвинутому** варианту подсистем `File Storage / Versioning` (не обязательно для MVP):
  - в распакованной области хранится только актуальная версия;
  - в процессе манипуляций допускается временное хранение 2–4 версий распакованными;
  - после истечения 24 часов с момента перевода версии в неактуальное состояние, версия должна быть добавлена в общий архив версий документа и удалена из распакованной области.
- В MVP допускается реализация **без** архивирования (все версии хранятся как отдельные файлы).

---

## 10.7 Обновлённая статусная модель и словарь терминов

- Словарь статусов и терминов фиксируется как часть контрактов системы и не должен трактоваться по-разному в разных модулях.
- Базовый набор статусов:
  - `new` (новый);
  - `on_review` (на проверке);
  - `needs_fix` (на доработке);
  - `approved` (утверждён);
  - `archived` (архивирован);
  - `invalid` (ошибочен: файл версии сохранён, но метаданные не извлечены или разбор завершился с ошибкой);
  - `trashed` (в корзине, выдержка перед удалением).
- `trashed` используется как удерживающий этап перед окончательным удалением.
- Финальное физическое удаление выполняется отдельно (ручным действием по правам или отложенной задачей по политике retention).

---

## 10.8 Влияние на архитектуру подсистем

- `Catalog` и `Versioning` обобщаются до `DocumentCatalog` и `DocumentVersioning`.
- `Metadata` проектируется с поддержкой нескольких извлекателей по типам документов (`MetadataExtractor` per `document_type`).
- `Workflow`, `Comments`, `Permissions`, `Exports` остаются общими и применяются к обобщённому документу.
- Для идентификации и алиасов добавляется отдельная подсистема `Identity/Matching` (или соответствующий сервисный модуль).

---

## 10.9 Зафиксированные UI-решения для MVP

- Базовый UI-подход: гибрид (два уровня подробности) — общая таблица документов + отдельная карточка документа при углублении.
- Главная таблица в MVP реализуется в максимальном варианте: core-поля + доступные метаданные + служебные поля + индикаторы комментариев/обсуждений.
- Основной сценарий согласования на первом этапе:
  - простые действия смены статуса;
  - есть вариант подтверждения с обязательным комментарием;
  - есть явный вариант "продолжить без пояснения" как осознанное исключение.

---

## 10.10 Политики разрешения конфликтов и удаления

- Срок удержания в `trashed` по умолчанию: 30 дней (после чего допускается физическое удаление по политике retention).
- Восстановление документа из `trashed` разрешено тем же ролям, которые имеют право отправлять документ в `trashed`.
- Если алиас имени файла совпадает сразу с несколькими документами:
  - автопривязка блокируется;
  - пользователь обязан вручную выбрать целевой документ.

---

## 10.11 Массовые операции и обязательный аудит MVP

- Массовые операции в MVP: только массовые действия `trashed` / `restore`.
- Прочие batch-операции (массовый export, массовая смена статуса, массовое скачивание ZIP) откладываются на следующий этап.
- Минимальный обязательный аудит на первом этапе:
  - загрузка новой версии;
  - смена статуса;
  - перевод документа в `invalid`;
  - удаление (перевод в `trashed`);
  - восстановление из `trashed`.
- Физическое удаление документа также обязательно логируется (действие администратора).
- Скачивание файлов в минимальный обязательный аудит MVP не включается.

---

## 10.12 Аутентификация и объём данных для MVP

- Аутентификация в MVP: только локальные учётные записи Django (создание пользователей администратором). LDAP/SSO — отдельным этапом после MVP.
- Данные с существующего Яндекс.Диска: однократный импорт (скрипт или мастер в админке), после чего источником истины считается только система; постоянная двусторонняя синхронизация с диском в MVP не входит.
- Коллизии и неочевидные привязки при импорте:
  - спорные файлы **не импортируются** автоматически;
  - такие случаи попадают в отчёт импорта как требующие ручной обработки (дозагрузка/ручное сопоставление).

---

## 10.13 Матрица прав и уведомления для MVP

- Матрица прав на этапе MVP — **минимальная** (остальные роли уточняются после первого релиза):
  - **кафедра / разработчик УП**: загрузка новых версий; перевод `needs_fix` → `on_review` (повторная подача); отправка документа в `trashed`;
  - **учебный отдел / рецензент**: переходы `on_review` ↔ `needs_fix`, утверждение `approved`; отправка документа в `trashed`;
  - **читатель**: только просмотр и скачивание (без смены статуса и загрузки);
  - **администратор**: полный доступ, в т.ч. настройки, пользователи, импорт; отправка в `trashed`; **единственная** роль с правом окончательного физического удаления после удержания в `trashed`.
- Уведомления (email, push, внешние интеграции) в MVP **не входят**: достаточно отображения статусов и истории в интерфейсе.

---

## 10.14 Стенд и тестовые данные PLX

- Способ деплоя (Docker, ВМ и т.п.) на первом этапе **не фиксируется**; в коде и схеме данных по умолчанию закладывается **PostgreSQL** как СУБД продакшена.
- Для снижения риска по формату PLX: в репозитории хранится набор эталонных `.plx` (без секретов) и тесты парсера на этих файлах.

---

## 10.15 Первая загрузка, ошибка парсера, корзина и окончательное удаление

- При **первой** загрузке файла новый документ создаётся со статусом `new`; перевод в `on_review` выполняется только явным действием пользователя с соответствующими правами (не автоматически при загрузке).
- Если разбор PLX не удался (битый файл, таймаут, неожиданная структура):
  - **версия файла сохраняется**;
  - на версии фиксируется признак неуспешного извлечения метаданных и текст/код ошибки для отображения в UI;
  - статус документа переводится в `invalid` («ошибочен»);
  - после успешного извлечения метаданных на **следующей** загруженной версии документ возвращается в `new` (дальнейший цикл — как обычно).
- Пока документ в `invalid`:
  - разрешены комментарии;
  - разрешены только операции `trashed` / `restore` из операций lifecycle/workflow;
  - остальные переходы workflow запрещены до появления успешной версии.
- Отправка в `trashed`: разрешена ролям **кафедра**, **учебный отдел** и **администратор**.
- В MVP **не вводится** ограничение «кафедра только свои документы»: при наличии роли действие допускается по любому документу (как и для учебного отдела); сужение по области видимости — отдельным этапом после MVP.
- Окончательное **физическое** удаление после истечения удержания в `trashed` (30 дней): **только администратор**.

---

## 10.16 Часовой пояс, отчёт импорта и лимиты файлов

- Часовой пояс для отображения дат в UI, для суффиксов имён архивных версий `_v...` и для отчётов: **Europe/Moscow**.
- Отчёт однократного импорта с Яндекс.Диска: **только экран результата в UI** (скачиваемый файл отчёта в MVP не обязателен).
- Разбор PLX в MVP выполняется синхронно в запросе загрузки.
- Таймаут разбора одного PLX в MVP: 10 секунд (после чего считаем ошибкой и переводим документ в `invalid`).
- Жёсткий лимит размера одного `.plx` в MVP **не вводится**; достаточно разумного таймаута и защиты от зависания при разборе.
