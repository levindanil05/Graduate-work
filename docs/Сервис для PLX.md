# Система для управления учебными планами (файлами plx)

## Что есть сейчас (as-is):

Яндекс-диск, в котором находятся актуальные и архивные версии (утверждённых учебным отделом) plx-файлов, в разделении по некоторому критерию (кажется, по факультету).  
Архив устроен примитивно: внутри каждой папки факультета создана папка АРХИВ, куда складываются, видимо, предыдущие версии файлов (скорее всего, не более одной — более старые удаляются безвозвратно).

Процедура добавления нового УП предполагает:  
1\) удаление старой архивной версии  
2\) перемещение текущей версии в архив  
3\) добавление (загрузка) нового файла в качестве актуальной версии.

… всё вручную.

### Проблемы:

\- не видны "внутренности" файла plx для предпросмотра, поиска и сортировки по информационным полям  
\- не видна (вся) история изменений и ответственные за эти изменения  
\- негде оставлять пометки и комментарии к отдельным файлам УП (какой актуальный, какой изменённый и почему, какой на удаление, и т.п.)  
\- нет резервирования хранилища  
\- процессы подачи на рассмотрение (пересылки файлов), самого рассмотрения, обсуждения, исправления и принятия (добавления в актуальные) не формализованы и нуждаются в автоматизации.

## Что требуется сделать (to-be):

Информационная система (веб-сайт), доступная как внутри учебного отдела, так и на факультетах вуза, включая филиалы, с разделением прав доступа (читатели, разработчики УП, комментаторы/рецензенты, управляющие, админы)

Структура сайта в простейшем варианте должна содержать таблицу со всеми файлами, загруженными в систему, давать их скачивать оттуда и закачивать новые файлы (автоматически попадают в таблицу) или обновлять имеющиеся (сопоставление по совпадению имени).

### Зеркала хранилища

Дополнительно могут подключаться хранилища-"зеркала", которые дублируют файлы основного хранилища.  
При этом второстепенные хранилища могут быть пассивными (исключительно бэкап туда) или активным, т.е. могут обновляться извне (как Яндекс-диск сейчас) — эти изменения надо отслеживать (по датам последнего изменения) и отражать на основном хранилище, (и аналогично на других второстепенных) \-- этот механизм распространения изменений нужно продумать аккуратно, чтобы избежать зацикливания такого автоматического распространения изменений.  
Это будет полезно для резервирования и дополнительного удобства доступа к файлам, а также для возможности “бесшовного” переезда с одного хранилища на другое.

### Управление дополнительными настраиваемыми полями

Концепция настраиваемых полей предполагает, что в основной таблице могут появляться сведения, которые изначально не предполагалось хранить, динамически, без изменения кода и вмешательства админов сервера.   
В панели управления сайтом должен быть раздел управления настраиваемыми полями. Будучи добавлено для объекта УП, доп. поле появляется для всех экземпляров УП в системе, и может получить своё специфическое значение для каждого экземпляра.  
Программно это не добавление колонки в основную таблицу объектов, а пара таблиц (сustom\_field)

Отображение извлечённых метаданных стоит делать через этот механизм, плюс должны быть доступны поля для свободного редактирования сотрудниками (политики, заметки, чат о файле)

### Хранение истории изменений.

Информация о файлах хранится в БД, включая все метаданные на время обновления.  
Также участник, который добавляет новую версию файла, т.е. загружает обновлённый документ, может написать комментарий наподобие cоmmit-сообщения, в котором описывает, что он сделал (Что было изменено?). Этот комментарий сохраняется вместе с метаданными и доступен участникам при просмотре версий.

### Жизненный цикл учебного плана и ролевая модель пользователей

Система предполагается многопользовательской для того, чтобы разные сотрудники вуза могли пользоваться ею для управления учебными планами в вузе на разных стадиях жизни УП.

Типичный пример стадий жизненного цикла УП (файла plx).

1. УП создаётся в недрах кафедры с помощью программы «Синяя звезда».  
2. УП Загружается в нашу систему со статусом: новый / ожидает проверки.  
3. Учебный отдел получает уведомление о том, что в системе появился новый план, ожидающий проверки.  
4. Учебный отдел скачивает файл, открывает его в синей звезде и проверяет.  
5. Предположим, в учебном плане были найдены ошибки, требующие коррекции.  
6. Учебный отдел оставляет комментарии к учебному плану, предложенному кафедрой, изменяют статус на “Ожидает исправления” (Опционально допускается при этом изменить учебный план, чтобы показать пример или внести некоторые изменения, которые кафедра не может сделать самостоятельно).  
7. Сотрудники кафедры получают уведомление о том, что пришли замечания к учебному плану.  
8. Возможен диалог кафедры с учебным отделом по поводу замечаний (все сообщения сохраняются, как в мессенджере, привязанными к этой версии УП).  
9. Сотрудники кафедры берут свой предыдущий файл, либо скачивают его из системы, (свой же или обновлённый в УО).  
10. Сотрудники кафедры изменяют учебный план, дорабатывают его, затем загружают файл в нашу систему снова, со статусом “ожидает проверки” и с историей, как вторая версия (после замечаний).  
11. Повторяются шаги 3 и 4\.  
12. Положим, на этот раз замечания были устранены, и учебный отдел изменяет статус плана на принято, актуально, утверждено (с точным термином позже определимся).  
13. Теперь актуальный УП может быть размещён на сайте (не напрямую, но материалы будут браться именно из этой актуальной версии), может быть использован для расчёта нагрузки и для других целей учебного отдела. Также кафедры могут при желании обратиться к системе, чтобы получить актуальную версию учебного плана для своих нужд.  
14. Далее, с появлением новой версии УП текущая архивируется.  
15. Удаление, конечно же, предусмотрено, \-- как только старых версий (сокращение истории с сохранением актуальной последней версии), так и целых учебных планов (включая актуальный).

# Зафиксированные решения (синхронизировано с roadmap)

Ниже перечислены решения, которые считаются согласованными и используются как требования к реализации.

## 1) Обобщённая модель сущностей

- Базовая сущность системы: **документ** (а не только учебный план).
- Учебный план — это тип документа: `document_type = "study_plan"`.
- Должна быть возможность добавлять новые типы документов без переписывания core-логики.
- Допускаются опциональные связи между документами разных типов (например, производные документы).
- Общие операции (версии, статусы, обсуждение версий, права, экспорт) применяются к обобщённому документу.

## 1.1) Термины пояснения и обсуждения

- У документа есть отдельное поле **пояснение документа** (общее пояснение), не привязанное к конкретной версии.
- Пояснение документа всегда отображается рядом с актуальной версией.
- Для коммуникации по версии используется **обсуждение версии**.
- Элемент обсуждения — **сообщение обсуждения**.
- Термин **комментарий** допускается для действий и описания изменений (например, комментарий к загрузке версии и к смене статуса).
- Комментарий действия/изменений не заменяет "пояснение документа" и не тождественен сообщениям обсуждения.

## 2) Дополнительные поля и хранение данных

- Используется гибридная модель:
  - поля для сортировки/фильтрации — в отдельной таблице (индексируемой);
  - более пассивные поля — в JSON-поле основной записи документа (`extra_data`).
- Для поля, хранимого в JSON, запись в таблице значений не создаётся, но создаётся запись в таблице описаний полей с типом хранения `inline`/`inner`.
- Набор индексируемых полей не должен быть жёстко зашит в код и должен расширяться без серьёзной переделки.

## 3) Идентификация по имени файла и алиасы

- Базовая идентификация УП строится по специальному имени файла (как в текущем процессе).
- Состав идентификатора в имени (факультет, направление, форма, кафедра, год и т.д.) не фиксируется жёстким шаблоном на уровне бизнес-логики.
- Поддерживаются алиасы имени-идентификатора (например, после транслитерации или других преобразований).
- Нужен механизм канонизации имени и сопоставления: `canonical_name + aliases`.
- Правило загрузки файла пользователем:
  - при добавлении файла система сначала пытается сопоставить его с существующим документом по имени/алиасам;
  - при совпадении — спрашивает пользователя, действительно ли он хотел привязать файл к найденной цепочке документа;
  - при отсутствии совпадений — сохраняет как новый документ (новая цепочка версий) без истории.
- Правило для действия `Загрузить новую версию`:
  - имя загружаемого файла должно быть совместимо с текущим документом (по `canonical_name + aliases`);
  - при несовместимости загрузка блокируется и показывается сообщение с требованием выбрать другой файл.

## 4) Идемпотентность повторной загрузки

- Повторная загрузка идентичного файла должна быть идемпотентной.
- Проверка идентичности делается по хэшу содержимого (предпочтительно `xxhash`).
- Если файл-дубликат уже существует:
  - новая версия не создаётся;
  - пользователь получает ссылку на совпавшую существующую версию;
  - по явному подтверждению пользователь может активировать найденную версию как "свежую" без повторного сохранения бинарных данных.

## 5) Статусы документа

- Фиксируется словарь статусов:
  - `new` — новый;
  - `on_review` — на проверке;
  - `needs_fix` — на доработке;
  - `approved` — утверждён;
  - `archived` — архивирован;
  - `invalid` — ошибочен (версия сохранена, метаданные не извлечены или разбор с ошибкой);
  - `trashed` — в корзине (выдержка перед удалением).
- Физическое удаление выполняется только после этапа `trashed` по политике retention и при наличии прав; **выполнять окончательное удаление может только администратор**.
- Срок удержания в `trashed` по умолчанию: 30 дней.

## 6) Версионирование и именование файлов

- Актуальная версия файла хранится под исходным именем (без суффиксов).
- Старые версии именуются с добавлением суффикса `_v{дата}` с детализацией при коллизиях:
  - `_vYYYYMMDD`;
  - `_vYYYYMMDD-HHMM`;
  - `_vYYYYMMDD-HHMMSS`;
  - при коллизии в ту же секунду: `(2)`, `(3)` и т.д.

## 7) История версий и архивирование

- История версий по умолчанию не удаляется.
- Целевая реализация хранения старых версий — архивирование в один сжатый архив на документ.
- Допускается эвристическая оптимизация упаковки, учитывающая высокую схожесть версий (ориентир 90%+).
- Это **продвинутый** вариант `File Storage / Versioning` (после MVP):
  - распакованной держим только актуальную версию;
  - временно (в процессе операций) допускается 2–4 распакованных версии;
  - через 24 часа после перевода версии в неактуальное состояние версия архивируется вместе с остальными и удаляется из распакованной области.
- В MVP допускается реализация **без архивирования** (все версии как отдельные файлы).

## 8) Разрешение конфликтов по алиасам

- Если загруженное имя-алиас соответствует нескольким документам, автопривязка блокируется.
- Пользователь должен явно выбрать целевой документ вручную.

## 9) UI-ориентиры для MVP

- Базовый формат интерфейса: гибрид — общая таблица + отдельная карточка документа при детализации.
- Главная таблица в MVP делается в максимальной комплектации: core-поля + метаданные + служебные поля + индикаторы обсуждения.
- Для workflow в MVP:
  - действия смены статуса остаются простыми;
  - поддерживается подтверждение с обязательным комментарием к действию;
  - доступен явный вариант "продолжить без комментария" как осознанное действие пользователя.

## 10) Правила восстановления из корзины

- Восстановление документа из `trashed` разрешено тем же ролям, которым разрешена операция отправки документа в `trashed`.

## 11) Массовые операции в MVP

- В MVP поддерживаются только массовые действия: отправка в корзину и восстановление из корзины.
- Массовая смена статуса, массовый экспорт и массовая загрузка ZIP откладываются на следующий этап.

## 12) Минимальный обязательный аудит (этап 1)

- Обязательно логируются:
  - загрузка новой версии;
  - смена статуса;
  - перевод документа в `invalid`;
  - отправка в `trashed`;
  - восстановление из `trashed`.
  - физическое удаление документа.
- Скачивание файлов в минимально обязательный аудит первого этапа не включается.

## 13) Аутентификация в MVP

- В первом релизе: только локальные пользователи Django; администратор заводит учётки. Интеграция с LDAP/Active Directory или SSO (OIDC/SAML) планируется после MVP.

## 14) Миграция с Яндекс.Диска

- В MVP: однократный импорт файлов (и при необходимости метаданных/структуры) из текущего хранилища; далее работа ведётся только в системе.
- Постоянное зеркалирование или двусторонняя синхронизация с Яндекс.Диском в объём MVP не входит.
- Коллизии и неочевидные привязки при импорте: спорные файлы пропускаются и отображаются в UI-отчёте импорта как требующие ручной обработки.

## 15) Матрица прав MVP (минимальная)

- **Кафедра / разработчик УП**: загрузка версий; после доработки — перевод `needs_fix` → `on_review`; отправка документа в `trashed`.
- **Учебный отдел / рецензент**: цикл проверки `on_review` ↔ `needs_fix`, утверждение в `approved`; отправка в `trashed`.
- **Читатель**: просмотр и скачивание без изменения данных.
- **Администратор**: полный доступ, пользователи, настройки, однократный импорт; отправка в `trashed`; **единственная** роль с правом окончательного физического удаления после удержания в `trashed`.

## 16) Уведомления в MVP

- Email, push и прочие внешние уведомления в MVP **не реализуются**; пользователи ориентируются на статусы, таблицу и историю в UI.

## 17) Стенд, СУБД и тестовые PLX

- Способ деплоя (Docker/ВМ) на первом этапе не фиксируется; в реализации по умолчанию ориентируемся на **PostgreSQL** как на СУБД для продакшена.
- Для парсера PLX: в репозитории — набор эталонных `.plx` без чувствительных данных + тесты разбора на этих файлах.

## 18) Первая загрузка, ошибка парсера, корзина и purge

- Новый документ при первой загрузке файла создаётся со статусом `new`; переход в `on_review` только по явному действию (не автоматически).
- Ошибка разбора PLX: версия **не отклоняется** — сохраняется файл, на версии — флаг/детали ошибки извлечения; статус документа — `invalid`. После успешного парсинга на следующей версии документ снова в `new`.
- Пока документ в `invalid`: разрешены сообщения обсуждения и только операции `trashed/restore`; остальные переходы workflow запрещены до успешной версии.
- В `trashed` могут отправить: кафедра, учебный отдел, администратор (в MVP **без** ограничения кафедры только «своими» документами). Физическое удаление после 30 дней в корзине — только администратор.

## 19) Часовой пояс, отчёт импорта и лимиты

- Даты в UI, суффиксы `_v...` и отчёты импорта — в часовом поясе **Europe/Moscow**.
- Результат однократного импорта: **страница отчёта в интерфейсе** (без обязательного экспорта файла в MVP).
- Разбор PLX в MVP выполняется синхронно в запросе загрузки; таймаут разбора одного файла: 10 секунд.
- Лимит размера `.plx` в MVP не фиксируется; задаётся разумный таймаут/защита от зависания при парсинге.

## 20) Набор экранов и кнопок для замыкания UI-цикла (MVP)

- **Вход**: `Войти`, `Выйти`, ошибки авторизации.
- **Главная таблица документов**:
  - фильтры по статусу/типу/метаданным;
  - строчные действия: `Открыть`, `Скачать`, `Загрузить новую версию`, `В корзину`/`Восстановить`;
  - массовые действия: только `В корзину` и `Восстановить`;
  - верхние кнопки: `Добавить файл`, `Импорт` (админ), `Обновить`.
- **Диалог добавления файла**:
  - сначала попытка привязки к существующей цепочке по имени/алиасам с подтверждением пользователя;
  - при дубле по хэшу: `Перейти к версии` / `Активировать как свежую`;
  - при отсутствии совпадений: создание нового документа без истории.
- **Карточка документа**:
  - в шапке всегда отображается **пояснение документа** (общее пояснение для актуальной версии);
  - вкладки `Версии` / `Workflow` / `Обсуждение` / `Метаданные`:
  - кнопки: `Скачать актуальную`, `Загрузить новую версию`, `Изменить статус`, `В корзину`/`Восстановить`.
  - `Загрузить новую версию` выполняет валидацию имени файла на совместимость с текущим документом; при несовместимости показывает ошибку и не создаёт версию.
- **Диалог смены статуса**: `Подтвердить`, `Подтвердить без комментария`, `Отмена`.
- **Корзина**: список `trashed`, `Восстановить`, `Удалить физически` (только админ).
- **Импорт (админ)**: запуск и отчёт (`импортировано`, `пропущено`, `ошибка`, `требует ручной обработки`).
- **Админ-раздел**: пользователи/роли, настройки хранилища, словарь дополнительных полей.

### Ограничения по статусам в UI
- В `invalid`: доступны сообщения обсуждения и операции `trashed/restore`; workflow-переходы недоступны.
- В `trashed`: недоступны редактирование/workflow; доступны `Восстановить` и (для админа) `Удалить физически`.
