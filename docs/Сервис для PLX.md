# Система для управления учебными планами (файлами plx)

## Что есть сейчас (as-is):

Яндекс-диск, в котором находятся актуальные и архивные версии (утверждённых учебным отделом) plx-файлов, в разделении по некоторому критерию (кажется, по факультету).  
Архив устроен примитивно: внутри каждой папки факультета создана папка АРХИВ, куда складываются, видимо, предыдущие версии файлов (скорее всего, не более одной — более старые удаляются безвозвратно).

Процедура добавления нового УП предполагает:  
1\) удаление старой архивной версии  
2\) перемещение текущей версии в архив  
3\) добавление (загрузка) нового файла в качестве актуальной версии.

… всё вручную.

### Проблемы:

\- не видны "внутренности" файла plx для предпросмотра, поиска и сортировки по информационным полям  
\- не видна (вся) история изменений и ответственные за эти изменения  
\- негде оставлять пометки и комментарии к отдельным файлам УП (какой актуальный, какой изменённый и почему, какой на удаление, и т.п.)  
\- нет резервирования хранилища  
\- процессы подачи на рассмотрение (пересылки файлов), самого рассмотрения, обсуждения, исправления и принятия (добавления в актуальные) не формализованы и нуждаются в автоматизации.

## Что требуется сделать (to-be):

Информационная система (веб-сайт), доступная как внутри учебного отдела, так и на факультетах вуза, включая филиалы, с разделением прав доступа (читатели, разработчики УП, комментаторы/рецензенты, управляющие, админы)

Структура сайта в простейшем варианте должна содержать таблицу со всеми файлами, загруженными в систему, давать их скачивать оттуда и закачивать новые файлы (автоматически попадают в таблицу) или обновлять имеющиеся (сопоставление по совпадению имени).

Упомянутые файлы УП должны храниться в настраиваемом расположении (на выбор администратора сервера):

### Зеркала хранилища

Дополнительно могут подключаться хранилища-"зеркала", которые дублируют файлы основного хранилища.  
При этом второстепенные хранилища могут быть пассивными (исключительно бэкап туда) или активным, т.е. могут обновляться извне (как Яндекс-диск сейчас) — эти изменения надо отслеживать (по датам последнего изменения) и отражать на основном хранилище, (и аналогично на других второстепенных) \-- этот механизм распространения изменений нужно продумать аккуратно, чтобы избежать зацикливания такого автоматического распространения изменений.  
Это будет полезно для резервирования и дополнительного удобства доступа к файлам, а также для возможности “бесшовного” переезда с одного хранилища на другое.

### Управление дополнительными настраиваемыми полями

Концепция настраиваемых полей предполагает, что в основной таблице могут появляться сведения, которые изначально не предполагалось хранить, динамически, без изменения кода и вмешательства админов сервера.   
В панели управления сайтом должен быть раздел управления настраиваемыми полями. Будучи добавлено для объекта УП, доп. поле появляется для всех экземпляров УП в системе, и может получить своё специфическое значение для каждого экземпляра.  
Программно это не добавление колонки в основную таблицу объектов, а пара таблиц (сustom\_field)

Отображение извлечённых метаданных стоит делать через этот механизм, плюс должны быть доступны поля для свободного редактирования сотрудниками (политики, заметки, чат о файле)

### Хранение истории изменений.

Информация о файлах хранится в БД, включая все метаданные на время обновления.  
Также участник, который добавляет новую версию файла, т.е. загружает обновлённый документ, может написать комментарий наподобие cоmmit-сообщения, в котором описывает, что он сделал (Что было изменено?). Этот комментарий сохраняется вместе с метаданными и доступен участникам при просмотре версий.

### Жизненный цикл учебного плана и ролевая модель пользователей

Система предполагается многопользовательской для того, чтобы разные сотрудники вуза могли пользоваться ею для управления учебными планами в вузе на разных стадиях жизни УП.

Типичный пример стадий жизненного цикла УП (файла plx).

1. УП создаётся в недрах кафедры с помощью программы «Синяя звезда».  
2. УП Загружается в нашу систему со статусом: новый / ожидает проверки.  
3. Учебный отдел получает уведомление о том, что в системе появился новый план, ожидающий проверки.  
4. Учебный отдел скачивает файл, открывает его в синей звезде и проверяет.  
5. Предположим, в учебном плане были найдены ошибки, требующие коррекции.  
6. Учебный отдел оставляет комментарии к учебному плану, предложенному кафедрой, изменяют статус на “Ожидает исправления” (Опционально допускается при этом изменить учебный план, чтобы показать пример или внести некоторые изменения, которые кафедра не может сделать самостоятельно).  
7. Сотрудники кафедры получают уведомление о том, что пришли замечания к учебному плану.  
8. Возможен диалог кафедры с учебным отделом по поводу замечаний (все сообщения сохраняются, как в мессенджере, привязанными к этой версии УП).  
9. Сотрудники кафедры берут свой предыдущий файл, либо скачивают его из системы, (свой же или обновлённый в УО).  
10. Сотрудники кафедры изменяют учебный план, дорабатывают его, затем загружают файл в нашу систему снова, со статусом “ожидает проверки” и с историей, как вторая версия (после замечаний).  
11. Повторяются шаги 3 и 4\.  
12. Положим, на этот раз замечания были устранены, и учебный отдел изменяет статус плана на принято, актуально, утверждено (с точным термином позже определимся).  
13. Теперь актуальный УП может быть размещён на сайте (не напрямую, но материалы будут браться именно из этой актуальной версии), может быть использован для расчёта нагрузки и для других целей учебного отдела. Также кафедры могут при желании обратиться к системе, чтобы получить актуальную версию учебного плана для своих нужд.  
14. Далее, с появлением новой версии УП текущая архивируется.  
15. Удаление, конечно же, предусмотрено, \-- как только старых версий (сокращение истории с сохранением актуальной последней версии), так и целых учебных планов (включая актуальный).  
16.

# Комментарии.

Все принятые решения нужно зафиксировать в папке docs.
Доп. поля для УП могут храниться В отдельной таблице для удобства сортировки и фильтрации, а более пассивные могут для компактности быть непосредственно в таблице с учебным планом, в "логически расширяемом" поле типа JSON. 
Нужно также заложить систему идентификации учебных планов по имени файла. Изначально они названы файлы по уникальному специальному имени, по которым сейчас они и отслеживаются (зашиты факультет, направление подготовки, форма обучения (Очная и т.п.), кафедра и год начала подготовки. Этот перечень не должен быть заложен жёстко в логику, это просто как пример и может меняться). Могут появляться синонимы имени-идентификатора, появившиеся после транслитерации или других подобных преобразований, то есть оно не всегда имеет только одну форму.
Идемпотентность на повторной загрузке: должна достигаться при помощи хэширования (можно взять популярный сейчас xxhash).
В дальнейшем возможно добавление в систему обработки других видов документа, например, экспортированные на базе учебных планов или несвязанные документы. Я не хочу, чтобы логика проекта была исключительно вокруг учебных планов, пусть это будут документы с типом «учебный план» и возможностью добавлять другие типы, которые также опционально могут быть связаны с документами других типов.
Ко многим видам документов применимы одни и те же операции, пусть они будут применимы к "обобщённому документу".
